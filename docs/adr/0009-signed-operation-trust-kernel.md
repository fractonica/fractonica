# ADR 0009: Signed operation trust kernel

- Status: Accepted
- Date: 2026-07-18
- Supersedes: The prototype identifier and actor-attribution decisions in
  [ADR 0007](0007-causal-operation-log.md)
- Implementation: Cryptographic verification, strict projection checking,
  capability-authorized node admission, and signed SQLite persistence are
  implemented. Pairing and non-loopback transport remain separate work.

## Context

The first causal-log slice used client-generated UUID operation IDs and treated
the local installation UUID as its actor. That was sufficient to validate
append, head, merge, and tombstone semantics on loopback, but a second node
could not prove authorship or detect an operation that had been changed by an
intermediary. A local database lifecycle is also the wrong identity for a
person-controlled device, automation agent, sensor, or peer node.

Fractonica needs one small cryptographic representation that can be verified
offline and reproduced by Rust, TypeScript, Swift, and constrained adapters.
It must retain the causal DAG without imposing a blockchain, global sequence,
trusted clock, or central account authority.

## Decision

An operation consists of an unsigned deterministic-CBOR payload and a
COSE Sign1 Ed25519 signature over that exact embedded payload. JSON is an API
projection only and is never signed.

### Identity types

The following textual encodings are canonical. Prefixes and hexadecimal case
are significant; malformed, uppercase, or unknown variants are rejected.

| Type | Canonical text | Binary value |
| --- | --- | --- |
| `ActorId` | `actor:ed25519:<64 lowercase hex>` | The actor's exact 32-byte Ed25519 public key. |
| `NodeId` | `node:ed25519:<64 lowercase hex>` | The node's exact, distinct 32-byte Ed25519 public key. |
| `SpaceId` | `space:<64 lowercase hex>` | 32 unpredictable bytes generated by a cryptographic random source. All-zero is invalid. |
| `OperationId` | `sha-256:<64 lowercase hex>` | SHA-256 of the exact unsigned payload bytes. |

`ActorId` and `NodeId` are self-certifying: their public key bytes are carried
by the identifier rather than resolved through a server-controlled account
table. The same private key MUST NOT be used for both roles. A node that authors
system operations has a separately generated actor key and requires the same
space capability as any other actor.

`InstallationId` remains an implementation-local identifier for one database
lifecycle. It MUST NOT appear in signed payloads, capability decisions, peer
credentials, or user-visible proofs of authorship. `SpaceId` identifies an
authorization and future replication namespace; knowledge of it grants no
permission.

### Unsigned operation payload

The payload is the deterministic [RFC 8949](https://www.rfc-editor.org/rfc/rfc8949.html)
CBOR encoding of this exact fixed-position array:

| Index | Logical field | Encoding and rule |
| ---: | --- | --- |
| 0 | domain | Text, exactly `org.fractonica.operation`. |
| 1 | protocol version | Unsigned integer, exactly `1`. |
| 2 | space | Byte string, exactly 32 `SpaceId` bytes. |
| 3 | actor | Byte string, exactly 32 `ActorId`/public-key bytes. |
| 4 | entity | Byte string, exactly 16 canonical UUID bytes. Nil is invalid. |
| 5 | schema | Lowercase ASCII token matching `[a-z0-9][a-z0-9._-]{0,63}`. |
| 6 | causal parents | Array of zero through 64 32-byte operation digests. |
| 7 | authorization | Array of zero through 64 32-byte capability-operation digests. |
| 8 | occurred time | Nonnegative Unix milliseconds through signed 64-bit maximum. |
| 9 | nonce | Byte string, exactly 16 bytes. |
| 10 | body | One schema-defined bounded canonical-CBOR value. |

Causal parents and authorization references MUST each be unique and strictly
ascending by their raw 32-byte digest. This ordering is only a canonical
encoding rule; it assigns no priority or chronology and does not change their
set semantics. Constructors may sort an input set, but decoders MUST reject a
wire payload that is unsorted or contains a duplicate.

The nonce distinguishes two intentional operations whose other logical fields
are equal. It SHOULD be generated randomly; deterministic importers MAY derive
it from stable source identity when collision resistance and domain separation
are documented. The nonce is not a replay token or secret.

The complete unsigned payload is bounded to 2 MiB, nesting depth to 32, and
each array or map to 4,096 entries. Deterministic encoding uses shortest integer
and length forms, preferred floating-point serialization, definite-length
containers, and bytewise deterministic map-key ordering. Duplicate map keys,
CBOR tags, invalid UTF-8, non-finite values not admitted by a schema, trailing
bytes, and alternative encodings of the same value are rejected. Every schema
MUST define its body-to-CBOR mapping and bounds; raw JSON object bytes or map
insertion order are never signing inputs.

### Operation digest

The operation digest is:

```text
SHA-256(deterministic_unsigned_payload_bytes)
```

The canonical textual `OperationId` adds the `sha-256:` prefix to the lowercase
hexadecimal digest. The ID is derived and MUST NOT be included inside the
unsigned payload. A receiver recomputes it before admission.

The signature is deliberately excluded from the operation digest. Re-encoding
or replacing a signature cannot create another logical operation, while any
change to actor, space, entity, schema, parent set, authorization set, claimed
time, nonce, or body creates a different digest. A causal parent therefore
identifies exact immutable payload bytes rather than a mutable row or local
cursor.

### COSE Sign1 envelope

The signed wire value is a tagged COSE Sign1 structure as defined by
[RFC 9052](https://www.rfc-editor.org/rfc/rfc9052.html):

- tag: exactly CBOR tag 18;
- protected header: exactly `{1: -8}`, selecting COSE `EdDSA` from
  [RFC 9053](https://www.rfc-editor.org/rfc/rfc9053.html);
- unprotected header: an empty map;
- payload: the complete embedded deterministic unsigned payload bytes;
- signature: exactly 64 bytes produced with Ed25519 as defined by
  [RFC 8032](https://www.rfc-editor.org/rfc/rfc8032.html);
- external authenticated data: the empty byte string.

The operation domain and version inside the protected payload provide protocol
domain separation. No authority, routing, or interpretation may depend on an
unprotected header. Receivers reject detached payloads, extra
headers, untagged structures, non-deterministic CBOR, other curves or
algorithms, and signatures whose key differs from the payload `ActorId`, even
if a generic COSE library would accept them.

Verification is performed in this order:

1. enforce the encoded-size bound and parse exactly one deterministic COSE
   Sign1 value;
2. validate the exact protected and unprotected headers;
3. parse and validate the deterministic unsigned payload and all field bounds;
4. recompute `OperationId` from the payload bytes;
5. reconstruct the Ed25519 verification key from `ActorId` and verify the COSE
   `Signature1` structure; and
6. validate parents, schema semantics, capabilities, and local admission
   policy before changing projections.

A cryptographically valid but unauthorized operation may be retained in a
bounded quarantine for later graph completion; it MUST NOT become an entity
head or mutate an authoritative projection.

## Capability boundary

The `authorization` list names the immutable grant operations on which the
actor relies. Signature verification does not imply authorization. Grant-chain
evaluation, revocation, space bootstrap, and QR pairing are defined in
[ADR 0010](0010-space-capabilities-and-pairing.md).

## External sources

Source-system values are never accepted as operations or interpreted as
signatures. An import agent maps them into the current schema and authors new
operations with its own explicit capability. It MAY preserve source IDs as
bounded provenance, but MUST NOT claim its signature proves who created the
source bytes. Source cursors and revisions never enter signed protocol fields.

## Consequences

- Any node can verify operation integrity and authorship without contacting a
  central service.
- Causal history becomes a Merkle DAG of operation payload digests.
- Multiple parents express an explicit merge without imposing a total chain.
- Actor keys, not installations or login rows, identify authors.
- Canonical CBOR and COSE fixtures become cross-platform release gates.
- Schema codecs must have one deterministic semantic-to-CBOR mapping.
- Key custody, capability state, pairing, and private-payload encryption remain
  separate concerns rather than being implied by a valid signature.
